設計方針

名前依存の解消方針

現在のスクリプトは職員名を内部識別子として使っているため、氏名変更、表記ゆれ、同姓同名、Excel 側の誤読に弱い構造です。この資料では、名前を表示用に戻し、内部では固定 ID を使うための改修方針を段階的に整理します。

現状
名前主キー
読込、引継ぎ、内部 schedule が名前キー
目標
ID 主キー
名前は表示と照合補助に限定
移行方針
段階導入
互換を残したまま移行
優先度
高
誤マッチ防止と将来保守性に直結

1. 現状の問題

現行コードでは Excel 行特定、固定勤務読込、指定休日、前月末引継ぎ、solver 結果の schedule まで名前をキーに使っています。

箇所 現在の識別方法 問題
Excel 行マッチング D列の氏名を正規化して照合 表記ゆれや誤読に弱く、同姓同名にも弱いです
固定勤務、指定休日 employee.name を辞書キーに結合 名前が変わると、過去帳票や target workbook との対応が切れます
前月末引継ぎ previous_tails.get(employee.name) 氏名変更時に月跨ぎの制約が途切れます
solver 内部 decision_vars[employee.name, day, shift] 内部主キーが表示名と一体化しており、設計上の責務が混ざっています
出力 schedule schedule[employee.name] 後段の検証とレポートも名前依存になります

2. 目標構成

内部は employee_id を使う

EmployeeConfig に employee_id を追加し、solver、schedule、fixed_assignments、specified_holidays、previous_tail のキーをすべて ID に揃えます。

名前は表示用と Excel 照合補助に限定する

Excel 側には ID がないため、照合時だけ名前や別名を使います。内部処理に入った後は ID に変換して扱います。

項目 現状 目標
職員設定 name, row が中心 employee_id, display_name, aliases, row
内部辞書キー name employee_id
Excel 照合 name の正規化一致 aliases を含む候補一致、最後に row フォールバック
レポート表示 name display_name

3. 推奨データ構造

推奨フィールド
最低限必要なのは employee_id と display_name です。Excel 名寄せの安全性を上げるなら aliases も持つのがよいです。

フィールド 用途 例
employee_id 内部主キー。変更しない固定値 akanecco-kanno-misato
display_name GUI、レポート、Excel 出力に使う表示名 菅野 実里
aliases Excel 側の別表記や旧姓対応 菅野実里, 菅野 実里
row 最終フォールバックの行番号 29

4. 段階移行の進め方

段階 変更内容 目的
第1段階 config に employee_id を追加し、重複チェックを入れる 内部キー移行の土台を作る
第2段階 Excel 読込時に 名前から employee_id へ変換する層を作る 外部入力と内部処理を分離する
第3段階 fixed_assignments、specified_holidays、previous_tail を ID キー化する 月跨ぎと workbook 読込の名前依存を除く
第4段階 solver の decision_vars と schedule を ID キー化する 内部主キーを完全に分離する
第5段階 レポートと GUI は display_name に統一する 利用者の見え方を変えずに内部設計を置き換える

5. 最小安全ライン

  • いきなり solver まで変えず、まずは config に employee_id を追加して内部辞書だけ移す方が安全です
  • Excel 側に ID を埋め込めない前提なので、aliases と row フォールバックはしばらく残す必要があります
  • 同姓同名の可能性があるなら、名前一致だけで処理を続行せず警告に落とすべきです

実務上の最小構成
最初の一歩としては、employee_id 追加、名前と ID の変換表追加、読込結果を ID キーへ変換、レポートだけ display_name 表示に戻す、の4点で十分です。これで名前依存の大半を切れます。

6. この順で着手するのが妥当

  1. config に employee_id と display_name を追加する
  2. 読込直後の固定勤務、指定休日、前月末引継ぎを ID キーへ寄せる
  3. solver と schedule を ID キーへ移す
  4. GUI と HTML レポートは display_name 表示へ統一する
  5. 最後に aliases と重複名警告を強化する
