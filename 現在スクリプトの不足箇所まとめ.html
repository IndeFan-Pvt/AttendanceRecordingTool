1. この資料の見方

根拠: generate_akanecco_shift.py の load_config、build_schedule_model、solve_schedule、validate_schedule と、generate_akanecco_shift_gui.py の入力UI。

  • 2 から 5 は、現在のコードがどこまで対応しているかを分類した現状整理です
  • 6 は、現場運用に影響が大きく優先して埋める不足を抜き出した要約です
  • 7 から 9 は、実装前に決める仕様、実装タスク、着手順を続けて見られる構成にしています
  • 実装済み ルールや機能が現在のコードで明示的に処理されている
  • 一部対応 コア処理はあるが、仕様の厳密さ不足、GUI不足、または解釈依存が残る
  • 未対応 現在のデータモデルや制約に直接の実装がない

2. 実装済みの項目

項目 状態 根拠 補足
南北の早番・遅番配置 実装済み build_schedule_model の required_per_day 南北ごとの early と late の人数制約を毎日付与している
夜勤は南北合計で1人 実装済み build_schedule_model の night_total_per_day 1日ごとの夜勤総数制約として実装されている
遅→早禁止、夜勤翌日の制限 実装済み build_schedule_model の隣接日制約 validate_schedule でも再検証される
最長5連勤、夜勤2連続、連休上限 実装済み add_window_constraint と各 streak 制約 work、night、rest、rest+special を別々に判定している
指定休日前日の夜勤禁止 実装済み EmployeeConfig.specified_holidays と build_schedule_model validate_schedule でも未検証や違反を出せる
月初への前月末勤務引継ぎ 実装済み previous_tail、read_previous_tail_from_workbook、with_generate_overrides CLI側は前月勤務表を自動探索、GUI側は手動指定も可能
個別の休数、勤務回数上限 実装済み min_counts、max_counts、min_rest_days、max_rest_days JSONで設定すればコア制約として反映できる
指定日の日勤人数 実装済み rules.day_requirements 日付ごとの min / max 制約として入れられる

3. 一部対応だが不足がある項目

項目 状態 今ある実装 不足している点
指定勤務日 一部対応 fixed_assignments と read_fixed_assignments_from_workbook コアにはあるが、GUIから職員別・日付別に入力する仕組みがない
指定休日 一部対応 specified_holidays を JSON から読める GUIから直接編集できず、運用は設定ファイル依存になる
夜勤回数の平等化 一部対応 fairness_night_spread による差分制約 「夜勤回数制限のない職員」だけを明示的に選ぶ仕組みはなく、夜勤可能者全体が対象になる
土日休みの平等化 一部対応 fairness_weekend_spread による差分制約 対象は employment == full のみ固定で、個別例外や対象者選択ができない
夜夜を1回は入れる配慮 一部対応 prefer_double_night による目的関数の優遇 保証ではなく優遇に留まるため、必ず1回入る仕様にはなっていない
通常の「日」をどこかに入れる条件 一部対応 一部社員に min_counts の「日": 1」を設定可能 ルールとして一般化されておらず、対象者定義も validate_schedule 側では曖昧なまま

4. 現在のコードで未対応の項目

項目 状態 不足の根拠 必要な拡張
人数不足でも途中経過を返す機能 未対応 solve_schedule は feasible / optimal でなければ RuntimeError を返すだけ 部分解の保持、違反付き出力、GUIでの暫定結果表示が必要
曜日ごとの勤務可能制限 未対応 allowed_shifts は職員単位の固定集合で、曜日別制限を持てない 職員ごとの曜日別 allowed_shifts か、日付別禁止制約が必要
「土日のどちらか休み」の週単位制約 未対応 build_schedule_model に週末ペア単位の休み制約がない 各土日ペアまたは各週末単位での休み制約モデルが必要
夜休の明示的なモデル化 未対応 夜休は shift_kinds に独立記号がなく、validate_schedule 側で近似判定しているだけ 夜休の定義確定、記号追加または派生状態としての一貫した実装が必要

重要

  • 夜休10日ルールは validate_schedule に検出処理があるが、生成時の制約には入っていない
  • 曜日限定遅番型の職員制約は、現在のデータモデルでは表現しきれない
  • 途中経過出力は、現行ロジックでは失敗時に何も返せない

5. GUI側で不足している点

根拠: generate_akanecco_shift_gui.py の ShiftGeneratorApp._build_ui と _run_generate。

項目 現状 不足
職員別条件の編集 GUIでは設定JSONファイルを選ぶだけ allowed_shifts、指定休日、回数上限、固定勤務を画面で編集できない
指定日条件の編集 day_requirements を画面から触れない イベント日の日勤人数指定をGUIから直接設定できない
制約違反時の再調整補助 失敗時はエラーメッセージとログ表示のみ 何を緩めればよいかの提案や、未充足条件の分類表示がない
暫定結果の表示 成功時のみ workbook と report を開く 失敗時の途中シフトや候補案を見せる仕組みがない

6. 優先して埋めるべき不足

未対応項目と一部対応項目のうち、ルール反映への影響が大きいものを先に絞り込んだ一覧です。

優先度 不足項目 理由
高 曜日ごとの勤務可能制限 理事長修正版ルールにある曜日限定遅番型を現状のままでは表現できないため
高 夜休の明示的なモデル化 夜休10日ルールの生成制約と検証ロジックが一致していないため
高 途中経過出力 運用要望に対して、現状は失敗時に何も返せないため
中 GUIでの職員別条件編集 コア処理はあっても運用が JSON 直編集前提になっているため
中 「通常の日」を入れる条件の一般化 現在は個別設定でしか代替できず、ルールとして明示されていないため

7. 実装前に決める仕様

未対応項目を実装可能にするため、先に判断が必要な仕様論点です。

対象 決めること 決まっていないと困る理由
夜休 独立した勤務記号にするか、夜勤翌日の休みから導出するか 夜休10日ルールを生成制約へ入れる方法と表示方法が変わるため
曜日別勤務制限 曜日ベースで持つか、日付ベースで禁止日を持つか 「金土日のみ遅番可」の設定形式が決まらないとデータモデルを拡張できないため
土日のどちらか休み 各土日ペアで判定するか、週単位で判定するか。夜勤明け休みを含めるか 制約モデルと検証ロジックの作り方が変わるため
途中経過出力 暫定シフトを返すのか、未確定セル付きで返すのか、違反一覧だけ出すのか solve_schedule の返り値と GUI の見せ方を先に決める必要があるため
夜勤平等化 対象者、許容差、必須か配慮か 現状は夜勤可能者全体が対象で、原文の「制限のない職員」と一致しないため
土日平等化 対象者、夜勤明け休みの扱い、許容差 誰の休みを比較するかが曖昧なままだと公平性ルールを固定できないため
通常の「日」確保 対象者、必要回数、他条件と競合したときの優先度 ルールとして一般化できていないため
指定勤務日 入力元を JSON、GUI、元勤務表のどれにするか。固定勤務の優先度をどうするか 運用方法が決まらないと UI と設定形式を揃えられないため
指定休日 入力元、GUI編集の有無、前月末情報不足時の扱い 検証だけでなく運用フローにも影響するため

9. 着手順

  • 1. 夜休の定義を決める
  • 2. 曜日別勤務制限の表現方法を決める
  • 3. 土日のどちらか休みの判定単位を決める
  • 4. 公平性ルールの対象者定義を決める
  • 5. 途中経過出力の形式を決める
  • 6. H1 から H5 のコア実装へ入る
  • 7. その後に GUI 入力範囲を広げる

この順にする理由

  • コア制約の定義が固まらないと、設定JSONもGUIも確定できない
  • 特に夜休と曜日制約は、現在のデータモデル拡張の起点になる
  • H1 から H5 は solver と設定モデルに直接影響するため、先に着手する価値が高い
  • 途中経過出力は、制約不成立時の扱いを根本から変えるため後回しにしすぎない方がよい

10. 要点まとめ

  • 現在のコードは基本配置と主要な回数制約までは対応済みで、未対応の中心は曜日依存条件、夜休、週末制約、途中経過出力です
  • 最初に固めるべきなのは、夜休、曜日制約、週末制約、公平性対象、途中経過出力の仕様です
  • 実装は H1 から H5 のコア拡張を先行し、その後に GUI 改善へ進む流れが自然です
