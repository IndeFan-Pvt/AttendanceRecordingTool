先に押さえること

この 3 種類の情報が揃うと実装しやすくなります。
  • 対象者: 全員なのか、一部職員だけなのか、除外条件は何か
  • 定義: 何を 1 回と数えるか、何を休みや通常日とみなすか
  • 例外運用: 常設ルールか、月ごとの上書きを許すか

現在のコードは、近い考え方の制約や検証はすでに持っています。ただし、原文どおりに実装するには対象集合と数え方が固定されていないため、そこを先に決める必要があります。

最小回答テンプレート

実装判断を進めるなら、まずは次の 6 行が埋まれば十分です。細かい運用差分はそのあと period_overrides へ落とせます。

そのまま埋める形式

  • No.9 対象者 / 除外条件 / 許容差 / 月別例外
  • No.10 対象者 / 最低回数 / 必須か配慮か / 例外月
  • No.11 hard 上限 / 4連勤回数上限 / 対象者 / 月跨ぎ判定
  • No.12 対象者 / 休み扱い記号 / 許容差 / 例外月
  • No.13 通常の日の判定方法 / 除外対象 / 対象者 / 必要回数
  • No.14 連鎖定義 / 月跨ぎ判定 / 対象者 / 閾値

補足

  • No.11, No.13, No.14 は定義が固まらないとコード変更方針そのものが決まりません
  • No.9, No.10, No.12 は主に対象者としきい値が決まれば実装しやすいです
  • 月別例外が多い場合は、共通ルールより先に period_overrides 運用方針を決めた方が安全です
実装順の目安
  • 先に No.11, No.13, No.14 の定義を固定
  • 次に No.9, No.10, No.12 の対象者としきい値を固定
  • 最後に常設設定と month override の切り分けを確定

実装面の注意

現在のスクリプトは、勤務表からは主に日別セルの勤務記号、指定休日、前月末情報を読んでいます。 そのため、右端の設定列を使う方式にする場合は、既存の勤務表読み取りとは別に「追加設定列を読む関数」を増やす実装が必要です。 ただし設計としては十分現実的で、特に職員ごとの対象フラグ系は config 手入力より運用しやすくなる見込みがあります。
