employee_id 設計案

akanecco_shift_config.json に employee_id を追加するための具体案です。互換を壊さない移行案を先に定義し、その後の最終形まで見通せる形にしています。

追加フィールド
2から3個
employee_id、aliases、必要なら display_name
互換方針
name 維持
第1段階では既存 name を残す
ID 形式
固定採番
名前由来ではなく再利用しない連番

fixed、休日、前月末を先に ID 化

1. 採用したい設計原則

  • ID は名前から自動生成しません。明示的に設定ファイルへ保存する固定値にします
  • 一度採番した employee_id は退職後も再利用しません
  • 現在は employee_id を内部キー、display_name を表示名、name を互換入力用として併用します

理由
名前由来 ID は旧姓変更や表記揺れのたびに揺れます。今回は名前依存をなくすのが目的なので、ID は意味より安定性を優先します。

2. 第1段階の JSON 形

現在の実装はこの段階を拡張し、display_name を正式な表示名として追加しています。name は旧設定との互換入力用です。

{
  "employee_id": "akanecco-001",
  "display_name": "笹谷 正人",
  "name": "笹谷 正人",
  "aliases": ["笹谷正人", "笹谷 正人"],
  "unit": "南",
  "employment": "full",
  "row": 6,
  "allowed_shifts": ["早", "遅", "日", "夜", "休", "特"],
  "min_counts": {"夜": 3, "日": 1},
  "max_counts": {"夜": 4},
  "min_rest_days": 6,
  "max_rest_days": 10,
  "fixed_assignments": {},
  "previous_tail": []
}

読み方
内部では employee_id を優先し、レポートや検証表示は display_name を使います。Excel 読込では aliases と row を使って employee_id へ解決し、name は旧設定ファイルを読めるよう残しています。

3. 最終形の JSON 形

次の整理候補は、互換期間が終わった時点で name を設定から外し、display_name に一本化することです。

6. period_overrides でのルール

重要
period_overrides 内に職員別設定がある場合、name ではなく employee_id で結び直す方針にすべきです。ここが name のままだと、base に employee_id を足しても月別上書きでまた名前依存が残ります。

{
  "period_overrides": {
    "2026-01": {
      "employees": [
        {
          "employee_id": "akanecco-008",
          "allowed_shifts": ["早", "遅", "日", "休", "特"]
        }
      ]
    }
  }
}

7. ロード時の互換ルール

  1. employee_id があればそれを採用する
  2. employee_id がなければ、移行期間だけ normalize(name) から暫定 ID を作る
  3. 同一 employee_id の重複は即エラーにする
  4. aliases の正規化値が他職員と衝突したら警告ではなくエラーにする
  5. Excel 読込は aliases 一致を優先し、最後に row 一致をフォールバックにする

8. この設計案で最初にやること

  1. akanecco_shift_config.json の各 employee に employee_id を追加する
  2. 表記差がある職員だけ aliases を追加する
  3. load_config で employee_id の重複チェックを入れる
  4. 読込結果の固定勤務、指定休日、previous_tail を employee_id キーへ変換する
